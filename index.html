二分法遍历数据库

背景

id name value create_time

工作中遇到这样一种情况, 数据库中存在如上结构的表,该表写入qps可达1000+, value字段部分记录值为0。现有需求每分钟发起定时任务, 获取过去某段时间(假设为过去30分钟至2分钟)value值为0的数据, 经过计算将其更新为其他数值,更新过程中若出现异常达到某阀值(假设阀值为30%,即更新100条若失败条数超过30条)则改次定时任务终止,等待下次定时任务调度时再更新value字段。

方案

首先,数据量较大,单线程执行效率过低,故考虑使用多线程:主线程查询数据库,将查询结果放入任务队列交给子线程执行。这样出现了一个问题:

  1. 若主线程采用普通分页查询,查询条件为value=0 and create_time > #{start_time} and create_time < #{end_time} limit #{from},#{pageSize}, 每查询一次页数加一, 则可能出现跳页现象, 因为主线程第二次查询时, 某些value为0的记录已更新为其他值。
  2. 考虑主线程等待所有自线程执行完成后再执行下次查询, 但会严重影响本次的调用的执行效率

  3. 考虑采用二分法进行查找, 主线程查询时首先获取某段时间段内记录条数, 若条数大于pageSize, 则获取起止时间的中间时间middleTime, 查询startTime->middleTime,middleTime->endTime的总条数, 若查询条数依旧大于pageSize, 则再进行时间段拆分直到总条数小于pageSize时获取当前时间段的所有记录。

批处理异步调用监控

背景

工作中遇到这样一种情况：
通过定时任务,每隔一分钟拉取database中的数据,并对数据进行异步处理,处理完成后要获取本次任务执行的成功数量/失败数量等等信息。
实施过程中遇到两个难点:

  1. 由于对数据的处理是异步执行的,要获取到执行结果首先想到的是通过Future来实现,但这种方式在仅需要获取执行成功/失败数量等简单信息时,显得对代码侵入过大。且该中情况出现在多处,代码修改工作量较大。
  2. 由于是定时任务执行,故可能存在本次任务执行未完成而新的任务又发起的情况。未避免这种情况就需要明确知道本次任务的执行完成时间。

方案

  1. 建立HealCount类,用于保存线程的执行结果

  2. 建立AsyncContext类,提供主线程及任务线程调用入口

  3. 建立AsyncParam类,对任务参数进行封装

  4. 建立MonitorRunnable类,对Runnable对象进行封装

  5. 建立DefaultMonitorRunnable对象,实现Monitor抽象类

  6. 测试

总结

该方案可在修改少量代码的情况下完成预期的目标。但由于计数是由AtomicInteger实现,在线程池中线程数量较多的情况下需考虑对性能的影响。

    线程池中使用ct1(AtomicInteger类型)记录线程池状态及线程池中当前线程状态两个信息

    • 线程池状态及其切换

execute

addWorker

拒绝策略

  • AbortPolicy (默认,抛出RejectedExecutionException)
  • DiscardPolicy
  • DiscardOldestPolicy (删除队列中最早的任务,并重新尝试execute当前任务)

总结

线程池执行逻辑: 

1. 判断当前线程池中当前线程数量,若小于corePoolSize则创建新线程且线程创建后直接运行当前任务;
2. 若不小于corePoolSize则向线程池任务队列中添加当前任务;
3. 若添加失败,则尝试创建新的线程且线程成功创建后直接运行当前任务;
4. 若线程池容量大于maxPoolSize则reject当前任务;

仅当任务队列满时,线程池才会创建大于corePoolSize数量的线程。故自定义线程池参数时应注意corePoolSize/maxPoolSize及队列的大小,避免队列过长且corePoolSize与maxPoolSize差距过大导致仅有corePoolSize的线程数工作,而maxPoolSize不生效。
