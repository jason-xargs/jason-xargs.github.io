JVM

内存划分

  • 程序计数器
    • 当前线程所执行的字节码的行号指示器
    • 通过改变计数器的值来选取下一条需要执行的字节码指令
    • 每条线程都需要有一个独立的程序计数器, 各条先策划功能之间计算器互不影响
    • 该区域是Java虚拟机规范中没有认定任何OutOfMemoryError情况的区域
  • Java虚拟机栈
    • Java方法执行的内存模型; 每个方法在执行的同时都会创建一个栈帧, 用于存储局部变量表、操作数栈、动态连接、方法出口等信息
    • 每个方法从调用直至执行完成的过程, 对应着一个栈帧在虚拟机栈中入栈到出栈的过程
    • 生命周期与线程相同, 也是线程私有的
    • 局部变量表
      • 存放了编译器可知的各种基本数据类型、对象引用。(long、double占用2个局部变量空间, 其余数据只占1个)
    • Java虚拟机规范中定义了两种异常状态:
      • 如果线程请求的栈深度大于虚拟机所允许的深度->StackOverflowError
      • 如果虚拟机栈可以动态扩展, 当扩展时无法申请到足够的内粗->OutOfMemoryError
  • 本地方法栈
    • HotSpot虚拟机直接把本地方法栈和虚拟机栈合二为一
  • Java堆
    • 如果对中没有内存完成实例分配, 并且堆也无法再扩展时->OutOfMemoryError
  • 方法区
    • 用于加载已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等
    • Non-Heap
    • 永久代
      • JDK1.7中, 将原来放在永久代的字符串常量池移除
    • 当方法区无法满足内存分配需求时->OutOfMemoryError
    • 运行时常量池
      • class文件中除了有类的版本、字段、方法、接口等描述信息外, 还有一项信息是常量池, 用于存放编译器生成的各种字面量和符号引用, 这部分内容将在类加载后进入方法区的运行时常量池中存放
      • 字节码生成技术/String.intern()方法, 可会将新的常量放入池中
      • 当常量池无法再申请到内存时, 会抛出OutOfMemoryError
  • 直接内存
    • 并不是虚拟机运行时数据的一部分
    • NIO可以使用Native函数直接分配堆外内存, 然后通过一个存储在Java堆中的DirectByteBuffer对象来作为这块内存的引用来进行操作
    • 避免了在Java堆和Native堆中来回复制数据

      对象的内存布局

  • 对象头(Header)
    • 用于存储对象自身的运行时数据
      • 哈希码(HashCode)
      • GC分代年龄
      • 锁状态标记
      • 线程持有的锁
      • 偏向线程id
      • 偏向时间戳
      • 这部分数据的长度在32位和64位虚拟机中分别为32bit和64bit(未开启压缩指针), 官方称之为’MarkWord’
    • 类型指针, 对象指向它的类元数据的指针
    • 对象位数组时, 对象头中还有一块用于记录数组长度的数据(虚拟机可以通过普通Java对象的元数据信息来确定对象的大小)
  • 实例数据(InstanceData)
  • 对象填充(Padding)
    * 对象的大小必须是8字节的整数倍, 当对象实例数据部分没有对齐时, 就需要通过对齐填充来补全

    引用

  • 强引用
  • 软引用
    • 在系统将要发生内存溢出异常之前, 把这些对象列进回收范围中进行第二次回收
  • 弱引用
    • 被弱引用关联的对象只能生存到下一次垃圾收集发生之前
  • 虚引用(幽灵引用/幻影引用)
    • 不会对齐生存时间构成影响, 但能在这个对象被垃圾回收器回收时收到一个系统通知

      垃圾收集算法

  • 标记-清除算法
    • 产生大量不连续的内存碎片
  • 复制算法
  • 标记-整理算法
  • 分代收集算法

    CMS

  • 运作过程
    • 初始标记
      • stop the world
      • 标记gc roots能直接关联到的对象
    • 并发标记
      • 进行gc roots tracing
    • 重新标记
      • 修正并发标记期间因用户程序继续运行而导致标记产生变动的那一部分对象的标记记录
      • 停顿时间比初始标记阶段稍长一些
    • 并发清除
  • 优点
    • 并发收集
    • 低停顿
  • 缺点
    • 对cpu资源敏感, 默认启动的回收线程数是(cpu数量+3)/4
    • 无法处理浮动垃圾, 可能出现ConcurrentModeFailure失败而导致另一次FullGc
      • 由于cms并发清理阶段用户线程还在运行, 伴随程序运行还有会新的垃圾不断产生
      • 由于垃圾收集阶段用户线程还需要运行, 需要预留足够的内存空间给用户线程使用。默认当老年代使用了92%时就会触发cms(-XX:CMSInitiatingOccupancyFraction), 若cms运行期间预留的内存无法满足程序需要, 就会出现ConcurrentModeFailure, 临时启用SerialOld收集器来重新进行老年代的垃圾收集
    • 基于标记-清除算法实现, 收集结束后会有大量的空间碎片产生。(–XX:UseCMSCompactAtFullCollection 开启内存碎片的合并整理过程 -XX:CMSFullGCsBeforeCompaction 设置执行多少次不压缩的FullGC后, 执行一次带压缩的, 默认为0, 表示每次进入FullGC时都会进行碎片整理)

      内存分配

  • 大多数情况下, 对象在新生代Eden区中分配。当Eden区中没有足够的空间进行分配时, 虚拟机将发起一次MinorGC
  • 大对象直接进入老年代(-XX:PretenureSizeThreshold, 另大于这个设置的对象直接在老年代分配)
  • 长期存活的对象将进入老年代(-XX:MaxTenuingThreshold, 对象晋升老年代的年龄阀值)
  • 动态对象年龄判定
    • 如果在Survivor空间中相同年龄的所有对象大小的综合大于Suvivor空间的一半, 年龄大于或等于该年龄的对象可直接进入老年代
  • 空间分配担保
    1. MinorGC前, 虚拟机首先检查老年代最大可用的连续空间是否大于新生代所有对象总空间
    2. 若条件1成立, 则本次gc可以确保是安全的
    3. 若条件1不成立, 虚拟机将查看HandlePromotionFailure设置值是否允许担保失败
    4. 若条件3成立, 则继续检查老年代最大可用内存是否大于历次晋升到老年代对象的平均大小
    5. 若条件4成立, 则尝试进行一次MinorGC
    6. 若条件3不成立或条件4不成立, 则进行一次FullGC

二分法遍历数据库

背景

id name value create_time

工作中遇到这样一种情况, 数据库中存在如上结构的表,该表写入qps可达1000+, value字段部分记录值为0。现有需求每分钟发起定时任务, 获取过去某段时间(假设为过去30分钟至2分钟)value值为0的数据, 经过计算将其更新为其他数值,更新过程中若出现异常达到某阀值(假设阀值为30%,即更新100条若失败条数超过30条)则改次定时任务终止,等待下次定时任务调度时再更新value字段。

方案

首先,数据量较大,单线程执行效率过低,故考虑使用多线程:主线程查询数据库,将查询结果放入任务队列交给子线程执行。这样出现了一个问题:

  1. 若主线程采用普通分页查询,查询条件为value=0 and create_time > #{start_time} and create_time < #{end_time} limit #{from},#{pageSize}, 每查询一次页数加一, 则可能出现跳页现象, 因为主线程第二次查询时, 某些value为0的记录已更新为其他值。
  2. 考虑主线程等待所有自线程执行完成后再执行下次查询, 但会严重影响本次的调用的执行效率

  3. 考虑采用二分法进行查找, 主线程查询时首先获取某段时间段内记录条数, 若条数大于pageSize, 则获取起止时间的中间时间middleTime, 查询startTime->middleTime,middleTime->endTime的总条数, 若查询条数依旧大于pageSize, 则再进行时间段拆分直到总条数小于pageSize时获取当前时间段的所有记录。

批处理异步调用监控

背景

工作中遇到这样一种情况：
通过定时任务,每隔一分钟拉取database中的数据,并对数据进行异步处理,处理完成后要获取本次任务执行的成功数量/失败数量等等信息。
实施过程中遇到两个难点:

  1. 由于对数据的处理是异步执行的,要获取到执行结果首先想到的是通过Future来实现,但这种方式在仅需要获取执行成功/失败数量等简单信息时,显得对代码侵入过大。且该中情况出现在多处,代码修改工作量较大。
  2. 由于是定时任务执行,故可能存在本次任务执行未完成而新的任务又发起的情况。未避免这种情况就需要明确知道本次任务的执行完成时间。

方案

  1. 建立HealCount类,用于保存线程的执行结果

  2. 建立AsyncContext类,提供主线程及任务线程调用入口

  3. 建立AsyncParam类,对任务参数进行封装

  4. 建立MonitorRunnable类,对Runnable对象进行封装

  5. 建立DefaultMonitorRunnable对象,实现Monitor抽象类

  6. 测试

总结

该方案可在修改少量代码的情况下完成预期的目标。但由于计数是由AtomicInteger实现,在线程池中线程数量较多的情况下需考虑对性能的影响。

ThreadPoolExecutor

初始化

  • 构造器

  • ct1
    线程池中使用ct1(AtomicInteger类型)记录线程池状态及线程池中当前线程状态两个信息

    • 线程池状态及其切换

execute

addWorker

拒绝策略

  • AbortPolicy (默认,抛出RejectedExecutionException)
  • DiscardPolicy
  • DiscardOldestPolicy (删除队列中最早的任务,并重新尝试execute当前任务)

总结

线程池执行逻辑: 

1. 判断当前线程池中当前线程数量,若小于corePoolSize则创建新线程且线程创建后直接运行当前任务;
2. 若不小于corePoolSize则向线程池任务队列中添加当前任务;
3. 若添加失败,则尝试创建新的线程且线程成功创建后直接运行当前任务;
4. 若线程池容量大于maxPoolSize则reject当前任务;

仅当任务队列满时,线程池才会创建大于corePoolSize数量的线程。故自定义线程池参数时应注意corePoolSize/maxPoolSize及队列的大小,避免队列过长且corePoolSize与maxPoolSize差距过大导致仅有corePoolSize的线程数工作,而maxPoolSize不生效。
